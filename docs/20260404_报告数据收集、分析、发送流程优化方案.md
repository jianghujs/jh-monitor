# 报告数据收集、分析、发送流程优化方案

## 背景

- 发送报告只能通过江湖面板发送，并且一个服务器一个服务器发送，容易被识别为垃圾邮件。
- 数据收集、数据分析、消息发送耦合在一起，容易受当时服务器运行状态、SSH 连通性和瞬时异常影响。
- 数据目前主要保存在面板本地 sqlite 中，没有统一沉淀到 ES 做集中管理、查询和追溯。

## 总体思路

将现有流程拆成三段：

1. 数据收集：远端主机定时生成原始数据，并统一通过filebeat收集到 ES。
2. 数据分析：扫描当天的原始数据，生成当天的日报，包括每个主机的报告和全部主机的概览报告。
3. 数据发送：从 ES 读取已经分析完成且校验通过的报告，再统一发送。

这样做的好处是：

- 采集、分析、发送解耦，链路更稳定。
- 报告以周期数据为准，不受发送当时机器状态影响。
- 原始数据和最终报告都能留痕，方便追溯、补算和重发。

## 一、新增配置功能

在 `route/templates/default/config.html` 增加“服务器报告配置”区域，统一管理报告链路所需配置。

### 配置项

- ES 相关配置，至少包含：
  - ES 地址
  - ES 用户名 / 密码（如启用鉴权）
- 服务器报告异常阈值统一配置，至少包含：
  - CPU 阈值
  - 内存阈值
  - 磁盘阈值
  - SSL 证书到期阈值

### 配置存储

- 配置存储方式参考现有 `data/notify.json`、`data/growth_alarm.json` 等 JSON 配置文件实现。
- 服务器报告异常阈值使用独立配置文件 `data/report_config.json` 保存。
- ES 相关配置使用单独配置文件 `data/es.json` 保存，与报告阈值配置分离管理。
- 配置文件不存在时自动初始化为 `{}`。
- 配置保存时优先写入 JSON；为了避免写入中断导致配置损坏，建议采用临时文件写入后再替换正式文件。
- 前端页面只负责配置录入与展示，实际保存、读取、校验由后端配置接口统一处理。

## 二、报告数据收集

### 目标

采集各主机报告所需的原始数据，统一写入 ES，作为后续分析的数据来源。

### 方案

- 需要收集两类数据：
  - 实时系统数据（如：CPU、内存、磁盘使用率等）
  - 备份日志类（如：xtrabackup备份记录、mysqldump 备份记录 等）数据。
- 实时运行数据可采用两种方式落 ES：
  - 方式 A：通过 ansible 定时远程执行采集脚本，脚本执行后直接写入 ES。
  - 方式 B ✨：在客户端起一个定时执行的采集进程，先写本地日志，再由 filebeat 采集并写入 ES。
- xtrabackup、xtrabackup 增量备份、backup 这 3 个数据直接由 filebeat 对接各自日志文件或 JSON 文件，再写入 ES。
- 原始数据只负责记录事实，不直接拼装最终报告。

### 安装与更新客户端定时采集任务

- 客户端定时采集任务的安装逻辑统一放到 `scripts/client/install/debian.sh`。
- 客户端采集采用“采集脚本 + cron 定时执行”的方式。
- 安装阶段需要完成采集脚本部署、cron 注册，并在安装完成后立即执行一次采集，确保安装完成后即可开始生成本地采集数据。
- 采集任务需要配置为开机后自动生效；采用 cron 时，通过系统 cron 配置保证机器重启后仍会按周期执行。
- 在客户端更新阶段，需要兼顾旧 cron 清理、脚本覆盖、权限修复、重新注册 cron，避免出现重复任务或残留旧脚本。
- 更新脚本需要具备幂等性：重复执行时不会重复注册同一条 cron，不会造成同一采集任务被重复触发。
- `scripts/client/install/debian.sh` 需要负责完成采集脚本安装、cron 写入、安装时立即执行一次采集脚本、更新后的 cron 重载与校验。
- 更新时若 cron 已存在，应按“清理旧 cron -> 覆盖程序/配置 -> 重新注册 cron -> 执行一次采集脚本 -> 校验输出结果”的顺序执行。

## 三、报告数据分析

### 目标

将当天的原始数据汇总分析，生成当天的日报，并写入 ES 的报告索引。报告分为两类：每个主机的详细报告，以及全部主机的概览报告。

### 方案

- 在`江湖云监控`增加独立的分析任务，每天凌晨1点执行。
- 从 ES 中读取某台主机当天的原始数据。
- 按统一规则进行汇总、异常识别、摘要生成，异常规则第一版对齐 `/www/server/jh-panel/scripts/report.py`。
- 先生成每个主机的详细报告，再汇总生成全部主机的概览报告。
- 生成最终报告后分别写入 `host-report-single` 和 `host-report-overview` 两个索引，作为发送的唯一数据来源。

### 产出内容

- 单主机报告：展示某台主机当天的资源、备份、网站和服务状态。
- 主机概览报告：汇总全部主机当天的整体情况，重点展示异常主机、异常项分布和整体运行摘要，并在尾部拼接当天全部主机的单机报告列表。

### 异常判断规则

以 `/www/server/jh-panel/scripts/report.py` 为准，主要规则如下：

- 系统资源阈值沿用当前通知配置：`cpu`、`memory`、`disk`、`ssl_cert`。
- CPU、内存、资源使用率按统计周期内平均值判断；监控点超过阈值时，按“5 分钟内只计 1 次异常”的方式统计 `overCount`。
- 磁盘使用率按各挂载点当前使用百分比判断，超过 `disk` 阈值即标记异常。
- 最后监控时间早于统计周期开始时间时，判定为“系统监控状态异常”。
- 网站证书异常沿用现有逻辑：非 `lets` / `acme` 证书在剩余有效期小于 `ssl_cert` 配置时记为异常。
- MySQL 主从状态异常沿用现有逻辑：`io_running != Yes`、状态记录不是当天产生、`delay` 为空或大于 0，均判定为异常。
- Xtrabackup 全量备份当天没有成功记录时，判定为异常。
- Xtrabackup 增量备份当天缺少全量或增量成功记录时，判定为异常。
- MySQL Dump 当天没有备份记录，或最后一次备份时间早于统计周期开始时间时，判定为异常。
- Rsyncd/实时备份沿用现有逻辑：已启用的实时同步或定时同步在当天没有成功同步记录时，判定为异常；`realtime_delays > 0` 记为告警提示。
- Keepalived 服务状态不是 `start` 时，判定为异常。
- 若当天没有任何异常摘要，则默认生成“服务运行正常，继续保持”。

说明：

- 单主机报告仍会展示网站、JianghuJS、Docker、MySQL 容量等明细。
- 第一版“是否异常主机”的判断先完全复用 `report.py` 的现有口径，后续如需扩大到站点停机、容器停机等规则，再单独补充。
- 分析任务读取异常阈值时，统一从新的“服务器报告配置”中取值，不再分散写死在任务代码中。

## 四、报告数据发送

### 目标

从 ES 中读取当天已经生成且有效的报告，再统一发送，避免发送时临时去主机上取数和计算。

发送范围控制为：只发送异常服务器的单主机报告，以及全部主机的概览报告。

### 方案

- 在江湖云监控增加独立的发送任务，按报告配置定时执行。
- 发送渠道固定为邮件，沿用现有通知配置和 `jh.notifyMessage` 邮件发送能力。
- 发送时只从 `host-report-single` 和 `host-report-overview` 读取已经分析完成且校验通过的报告。
- 单主机报告只发送当天识别为异常的服务器。
- 每天固定发送一份全部主机的概览报告，且该报告尾部附带当天全部主机的单机报告列表。
- 报告内容中保存 HTML，发送时优先直接使用已生成的 HTML 内容。

- 发送任务、分析任务、采集任务访问 ES 时，统一读取同一套 ES 配置，避免多处维护。
